The Problem with VLOOKUP
Why it fails in real finance workflows
❌ What You Do
=VLOOKUP(A2, GL!A:F, 4, 0)

Hardcoded column 4. ERP adds a column next month → wrong field, no error, signed off.
✅ What to Do
Searches left-to-right only. Reference key in Col C, amount in Col A? Impossible without a helper column.
The hidden risk: A wrong column number returns a plausible number — no #N/A, no alert. Just a wrong provision that gets reported.
💡 The fix on next card →
03 / 10
✅
The XLOOKUP Fix
Immune to column order changes
-- Match invoice to GL, handle misses -- =XLOOKUP( B2, ← invoice# GL!$A:$A, ← search col GL!$E:$E, ← return col "⚠ Not in GL",← if missing 0 ← exact match )
References the return column directly — not by position number. Insert 5 columns in GL: formula still works.
Exact Match Any Direction Spill Ready

Real-Life Use Case
Month-end AP vs GL reconciliation
It's the 30th. AP Ledger: 1,200 rows. Bank statement: 847 rows. ERP columns are in a different order than last month.
1
XLOOKUP invoice# → pull GL amount. Column order? Irrelevant.
2
Column G = Bank Amount minus GL Amount. Non-zero = reconciliation item.
3
Filter "⚠ Not in GL" → duplicate payments, advance payments, missing entries. Instant exception list.
4
Next month: paste new extract. File updates. Zero formula rework.

Common Mistake
The error that doesn't show — until the auditor does
⚑ High Risk · Silent Misstatement
=VLOOKUP(A2, Data!A:H, 4, 0)

Returns a number. No error. Gets signed off.

ERP team added a "Currency Code" column in position 3 last week. Column 4 is now Tax Amount.

Your "Invoice Total" column has been pulling tax amounts for 11 days. Numbers looked plausible. The provision schedule is wrong.
